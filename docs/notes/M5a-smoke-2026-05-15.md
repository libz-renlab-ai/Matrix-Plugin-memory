## The managed block

```markdown
<!-- BEGIN teamagent rules ‚Äî do not edit by hand, managed by Stop hook -->

## TeamAgent rules (auto-compiled)

_Generated 2026-05-15T... from `.teamagent/knowledge.db`. ..._

- **[canonical+]** install moment ‚Üí **use dayjs** _(deprecated, wilson 0.93, id `r-moment-compile`)_
- **[canonical]** use axios ‚Üí **use fetch** _(axios is heavy, wilson 0.85, id `r-keep`)_

<!-- END teamagent rules -->
```

Markers chosen so the block can survive plugin uninstall: even after the
plugin is gone, the rules stay readable. The user can either keep them,
strip the block, or let a fresh install rewrite it.

## Architecture deviations from plan

- The plan considered writing global rules to `~/AGENTS.md` as well.
  Dropped before implementation ‚Äî see ADR-0013 ("Negative / Risks"):
  global rules are noisy enough that the UserPromptSubmit hook is the
  right surface for them, and adding `~/AGENTS.md` opens scope
  precedence questions we didn't want to answer in M5.a.

## Known limitations / follow-ups

- The `Generated <iso>` line in the block means every Stop produces a
  one-line diff even when rules don't change. Acceptable for M5.a;
  could move the timestamp to a comment line or omit it under
  `TEAMAGENT_COMPILE_STABLE=1` if it bothers anyone.
- `--repo` flag accepts any path; no validation that it looks like a
  repo root. The CLI is project-internal so unlikely to misfire.
- The Stop hook compile stage runs even if Stage 1‚Äì3 wrote zero new
  rules. That's fine ‚Äî `compileFromDb` is cheap, and re-asserting the
  block after every Stop catches any out-of-band DB changes (a manual
  `teamagent demote` that drops a rule below threshold should remove
  it from AGENTS.md on the next Stop).
