## Tuning: semantic exception threshold

Initial threshold of **0.78** matched the design's `Î¸_sem = 0.78`, but the
override-flow integration test exposed a false-positive: with e5-small,
cosine for two unrelated short queries baselines around **0.79â€“0.80**, so
`cos("npm install moment", "in test fixtures") = 0.795` was wrongly
treated as a paraphrase match.

Verified empirically inside the worktree:

| query Ă— condition | cosine |
|---|---|
| `npm install moment inside the test suite` Ă— `inside automated test fixtures` (M4 positive) | **0.8475** |
| `npm install moment in test fixtures` Ă— `in test fixtures` (M3 positive) | **0.888** |
| `npm install moment` Ă— `in test fixtures` (false-positive) | 0.795 |
| `npm install moment` Ă— `inside automated test fixtures` (false-positive) | 0.799 |
| `help me debug this code` Ă— `in test fixtures` (false-positive) | 0.797 |

Raised `EXCEPTION_SEMANTIC_THRESHOLD` to **0.82**, leaving â‰Ą0.02 margin
both ways. This is distinct from the `Î¸_sem = 0.78` used for the L2
rule-matching layer, which compares full rule embed_text (Adopting moment.
Use dayjs. deprecated.) â€” that input is longer and more distinctive, so
the same threshold doesn't generalize. Documented inline at
`hooks/lib/match.cjs:7` and folded into ADR-0012.

## Architecture deviations from M4 plan

- Plan called for eager query embedding in `runMatch`. Changed to a lazy
  thunk after the unit test `runMatch async â€” fast-path bypasses embedding`
  caught the regression (model load = 60s on cold path). Now Layer 1
  embeds only if a hit's rule has an embedded exception; Layer 2 forces it
  as designed. Net result: zero behavioral change, but fast-path-only
  matches stay off the embedding hot path.

## Known limitations / follow-ups

- Project precedence only applies to PreToolUse and UserPromptSubmit. The
  Stop pipeline currently writes new rules only to global.db (no scope
  decision). When a project-tier rule is decided in M5+, the compile
  pipeline will need the same precedence consideration.
- Auto-classify uses a fixed `AUTO_CLASSIFY_AFTER = 3`. ADR-0010 keeps the
  user in the loop by default, so the threshold is intentionally a small
  constant rather than a per-rule config. If a user reports churn, exposing
  it via `teamagent config` is a clean future change.
- Semantic exception is single-vector cosine, no pooling across multiple
  exceptions for the same rule. With <10 exceptions per rule the linear
  scan is cheap, so this is unlikely to need rework before M6.
