## End-to-end override-flow demo (automated)

The `override-flow.test.cjs` integration test reproduces the full DESIGN §8 flow:

1. Insert a `canonical+` rule (`r-moment`).
2. PreToolUse on `npm install moment` → `deny` (block tier).
3. PostToolUse on the same command with `exit_code=0` (user "retried") → `override_detected` event logged.
4. `teamagent classify <event_id> b --condition "in test fixtures"` → exception saved.
5. Next PreToolUse with query `npm install moment in test fixtures` → silent pass (exception applied).
6. PreToolUse with query `npm install moment` (no exception trigger) → still `deny` (rule unchanged in general).

Both `classify b` (context-specific exception) and `classify a` (rule-wrong demote) covered.

## Known limitations / follow-ups

- **Exception match is literal substring + token presence**, not semantic.
  M4 may add embedding-based exception matching when the condition text is
  more abstract (e.g., "in test code" vs "in __tests__/" path).
- **One override prompt per session per rule**: if user ignores the prompt
  for several turns, M1 plan said "3 turns then auto-classify as (a)"
  (DESIGN §8.3). Not implemented in M3 — currently the override stays
  unhandled until the user replies or `teamagent classify` is run.
  Add an automatic timeout in M4.
- **`extractCondition` uses `claude -p`** with haiku — same cost / failure
  profile as Stop hook extract. Only invoked by the `mute-rule` skill
  (M3 doesn't auto-invoke it; humans take that branch).
